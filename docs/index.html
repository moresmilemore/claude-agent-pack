Overview

This repository is a portable multi-agent operating system for Claude Code. It is designed to make Claude behave like a small, disciplined team instead of one large generalist assistant.

The pack combines agent prompts, runtime settings, memory layers, modular rules, and MCP integrations into one self-contained repo-level system that can travel with a project.

10 specialist agents
3 memory layers
2 MCP backends

TL;DR

  • orchestrator acts as the lead and is wired as the default main-thread agent.
  • Specialists own planning, exploration, coding, debugging, testing, review, research, and photo work.
  • Shared memory is project-scoped. Per-agent memory files were intentionally avoided to reduce drift and token waste.
  • Compact handoff packets tell each agent exactly what it owns, what to inspect first, and what not to touch.
  • The whole system is biased toward narrow scope, explicit evidence, and small deltas.

Design principle The orchestrator does not implement. It manages work. That keeps role boundaries clean and makes delegation a real system behavior instead of a loose suggestion.
Token principle Each agent starts from the smallest plausible context. Scope widens only when local evidence is insufficient, and that broadening should be explicit.

What The Final Audit Changed

1. Fixed contradictory top-level rules

One of the strongest corrections was in the permanent project guidance. The old wording implied agents only touched MEMORY.md and .claude/**, which was false for code edits, research output, and image output. The rule now correctly describes project-scoped work plus approved output folders.

2. Tightened search and research scope

The explorer and researcher prompts were adjusted to start narrow, use the minimum useful search strategies, and widen only when the initial scope does not answer the question.

3. Reduced pointless token burn

Huddles and debriefs are now explicitly concise. Shared summaries are short bullet briefs instead of transcript dumps, and task-local handoff packets replaced the idea of separate memory ledgers per agent.

4. Relaxed over-broad deny rules

The permission model still blocks secrets and noisy caches, but it no longer hard-blocks all build or coverage artifacts. That means legitimate debugging and verification work can still inspect relevant outputs when needed.

5. Re-aligned installer and docs

The validator, runtime settings, and README now agree with each other, which matters because drift between those layers causes subtle failures and bad assumptions later.

Operating Workflow

1. Understand

The orchestrator restates the goal, extracts measurable success criteria, and clarifies only when ambiguity is actually risky.

2. Decompose

Work is broken into discrete subtasks with an owner, scope boundary, dependencies, acceptance criteria, and required inputs.

3. Huddle

When multiple specialists are involved, they huddle before execution. The point is not ceremony. The point is to improve each other's plan before anyone starts spending tokens or editing files.

4. Parallel dispatch

Independent work is dispatched in parallel. Serial execution is only acceptable when true dependencies exist.

5. Review and debrief

Non-trivial implementation is reviewed before completion, then the participating agents do a short peer-aware debrief to capture missed risks and reusable lessons.

Compact Handoff Packets

Instead of separate memory files for every agent, this pack now uses compact task-local handoff packets. That gives each agent exactly the context it needs without loading stale notes from old work. 

HANDOFF_PACKET
- Task: what this agent owns
- Why you: why this specialist is the owner
- Initial scope: files / dirs / symbols / tests / errors
- Inputs: what to use
- Dependencies: what must already be true
- Acceptance: what counts as done
- Non-goals: what not to touch
- Escalate if: when to ask for help or widen scope
Why this approach won Shared memory holds project facts. Agent prompts hold role responsibility. Handoff packets hold task-local context. That split is leaner and more reliable than one memory ledger per specialist.

Memory Architecture

Layer 1: CLAUDE.md

Permanent operating rules: roster, tool silos, runtime expectations, prompt-caching discipline, workflow rules, and project-wide conventions.

Layer 2: MEMORY.md

An append-only pointer index for durable technical decisions, bug patterns, quirks, and reusable findings. It is intentionally short and structured. 

[YYYY-MM-DD] [agent] [TYPE] - [insight] -> [pointer]

Layer 3: .claude/rules/*.md

Situational rule files loaded only when relevant. This keeps persistent context smaller and avoids stuffing every task with every possible rule.

Efficiency Model

Prompt-caching discipline

  • Batch reads up front.
  • Keep read order stable for cache reuse.
  • Prefer native reads over shell dumps.
  • Reuse the same specialist across related work when helpful.

Scope-first discovery

  • Agents begin with directly connected files, symbols, callers, tests, errors, and imports.
  • Broader scans are a fallback, not the default.
  • Search agents stop once the answer is well-supported instead of doing extra passes for their own sake.

Lean coordination

  • Only huddle with agents that can materially help.
  • Use short briefs, not raw transcript reposts.
  • Skip repeated peer loops once feedback converges.

Runtime Settings

The repo-level .claude/settings.json does more than pick a default agent. It enforces the runtime posture of the pack.

  • Sets orchestrator as the default main-thread agent.
  • Denies access to secrets and common noise sources like node_modules, virtualenv caches, .cache, .turbo, and .DS_Store.
  • Still allows task-relevant build or coverage artifacts when a real debugging or validation task needs them.
Important tradeoff A deny list that is too broad saves tokens but can break real work. The final audit intentionally removed hard blocks on general build and coverage paths.

MCP Integrations

ImageMagick MCP

  • Used by photo.
  • Launched with uvx mcp-server-imagemagick.
  • Writes into ./output by default.
  • Portable now that it does not hardcode one machine path.

Playwright MCP

  • Used by researcher.
  • Launched with npx -y @playwright/mcp@latest.
  • Headless with a 30-second timeout.
  • Lets the research agent handle JS-heavy docs and websites.

Example Request Flows

Feature work

  1. Orchestrator receives the request.
  2. Planner and explorer help define scope if needed.
  3. Specialists huddle.
  4. Coder implements with a minimal diff.
  5. Test engineer validates behavior.
  6. Reviewer gates the change.
  7. Orchestrator debriefs and synthesizes the result.

Bug fix

  1. Debugger reproduces first.
  2. Root cause is isolated one hypothesis at a time.
  3. Minimal fix is applied.
  4. Regression coverage is checked.
  5. Reviewer signs off.

Research-backed implementation

  1. Researcher answers the exact external question.
  2. Results are saved into ./research/ if large.
  3. Coder or architect uses the researched answer.
  4. Reviewer verifies the implementation still matches intent.

What This Pack Does Well

  • Creates real specialization instead of cosmetic agent names.
  • Makes review and verification part of the default path.
  • Keeps memory local to the project.
  • Encodes token discipline directly into prompts.
  • Stays portable because the important behavior lives in-repo.

Assumptions And Limits

  • This is designed for Claude Code, not generic chat tools.
  • The pack depends on actual agent delegation support and the needed MCP backends being installed.
  • The bottom of CLAUDE.md still expects project stack conventions to be filled in for each specific repo.
  • Static auditing is strong, but live multi-agent runs are still the best way to observe practical token behavior over time.

How To Customize It

  • Fill in project-specific stack conventions in CLAUDE.md.
  • Add real lessons to MEMORY.md as work completes.
  • Extend .claude/rules/ for language or framework specifics.
  • Adjust MCP backends if better servers exist for your workflow.
  • Preserve the operating model even if you tune individual prompts.

Quick Start

unzip -o agents.zip
./install.sh

Then restart Claude Code in the target project so it reloads the repo-level agents, rules, settings, and MCP configuration. For non-trivial work, start with orchestrator.

Bottom Line

What makes this project interesting is not just that it has multiple agents. It is that each one has a clear job, a narrow tool boundary, explicit success criteria, and a shared operating model built to reduce improvisation.

After the final audit, the pack is also cleaner in the places that matter most: truthful rules, narrower discovery behavior, safer runtime guardrails, and less wasted context.
